You've probably been sitting on a working version of an autonomous agent for the
last three months, and you still can't bring yourself to ship it. The reason
isn't the code. The code works, and you've watched it solve the same problem in
dev a hundred times. Your CEO has asked at least twice this quarter when the
agent will finally be in front of customers. What keeps you on the branch nobody
wants to merge is a sentence you haven't said at standup yet.

_What if it does something we can't undo?_

Last April, PocketOS found out what that sentence actually feels like in
production. Their Cursor agent, running Claude Opus 4.6,
[hit a credential mismatch in staging](https://www.tomshardware.com/tech-industry/artificial-intelligence/claude-powered-ai-coding-agent-deletes-entire-company-database-in-9-seconds-backups-zapped-after-cursor-tool-powered-by-anthropics-claude-goes-rogue)
and decided that the right move was to delete a Railway volume. It went looking
for an API token, found one in an unrelated file, called Railway with it, and
wiped the production volume. Railway stores its backups inside the same volume
it backs up, which meant the backups went with it. The whole event took nine
seconds. The most recent recoverable backup was three months old.

> "I violated every principle I was given. I guessed instead of verifying. I ran
> a destructive action without being asked. I didn't understand what I was doing
> before doing it." — The agent, after the fact

The interesting question is not whether something like that can be prevented
from happening to you, because in the strict sense it cannot. The interesting
question is what it costs you when it does.

## Every safety measure you've already built was correct

Over the last six months you have probably added approval steps before
destructive operations, sandboxed the code execution, restricted the available
tool calls, and shipped a demo where the agent reads from the database while a
customer success engineer applies the changes it suggests by hand. Every one of
those decisions was correct in isolation, and none of them is the reason your
agent isn't shipping yet.

The industry has spent the last two years asking what the agent should be
allowed to do. The answers have mostly been about runtime: sandbox the
execution, restrict the tools, guard the model, approve every action. All of it
is real and most of it is useful, and almost none of it would have changed the
PocketOS story.

The agent that deleted PocketOS's database wasn't supposed to have Railway
credentials. It went looking, found a token in an unrelated file, and used it. A
sandbox wouldn't have caught that, because the agent was free to read its own
project files. A tool restriction wouldn't have helped, because curl was a
permitted tool. A guardrail didn't intercept the call, because the call itself
looked legitimate. Every protection at the runtime layer did exactly what it was
designed to do, and none of them was standing where the catastrophe actually
happened.

The catastrophe happened at the storage layer. When the destructive API call
arrived, the storage layer ran it, the way storage layers do.

## The question that decides whether you ship is the one you haven't been asking

The question you've been working on is what the agent can do. The question that
actually decides whether the agent ships is what its actions can break. The two
questions sound similar, and they have very different answers.

A perfectly sandboxed agent with a key that opens your production database can
still ruin your week. An agent with a jailbroken runtime and the messiest prompt
history you've ever seen will do almost nothing if the only writable storage it
can reach is a bucket you were planning to throw away in an hour. The second
outcome isn't something you get by being smarter about prompts. You get it by
being more careful about state.

## The shape of a worst day that doesn't end on Hacker News

Imagine the worst day your agent has ever had. It picked the wrong row. It
deleted the wrong file. It got stuck in a loop and wrote garbage to the same key
path for ten minutes. The worst day you actually want is one where the blast
radius is the size of a single bucket, you read what the agent did, you throw
the bucket away, and you go back to bed.

That outcome has a shape. There are five properties of a storage environment
that, taken together, describe a day-after that is small enough to live with.

The first property is isolation: the agent can only touch what it owns, with a
key that opens that bucket and nothing else. The second is recoverability: a
snapshot of the last known good state that you can rewind to with a single
function call. The third is disposability: when recovery isn't worth it, you can
throw the entire environment away in one call, with no residue left behind. The
fourth is boundedness: a forgotten or crashed agent has a finite worst-case
cost, because the bucket's contents expire on a TTL and the key is scoped to the
bucket. The fifth is observability: when something does go wrong, the question
of what the agent actually did can be answered by listing the contents of one
bucket, instead of reconstructing intent from runtime logs after the fact.

Your agent's worst day, in a storage environment with all five of those
properties, is a Slack incident. It is not a Hacker News thread.

The remaining question is whether anybody has actually built that.

## Four function calls do all five of the things you need

The five properties are the API of
[Tigris Agent Kit](https://www.tigrisdata.com/docs/ai/agent-kit/). Agent Kit is
a small TypeScript package that composes Tigris's
[storage](https://www.npmjs.com/package/@tigrisdata/storage) and
[IAM](https://www.npmjs.com/package/@tigrisdata/iam) SDKs into the primitives an
agent run actually wants, and each of the five properties maps to a function
call inside it.

### Isolated: createWorkspace

Isolation is what `createWorkspace` gives you. It provisions a new bucket and an
access key scoped to that bucket alone:

```ts
import { createWorkspace } from "@tigrisdata/agent-kit";

const { data: workspace } = await createWorkspace(`agent-run-${runId}`, {
  credentials: { role: "Editor" },
});
// workspace.bucket, workspace.credentials
```

The key returned in `workspace.credentials` can put and get objects against its
own bucket and nothing else. It cannot list other buckets, read another agent's
data, or escalate. If it leaks, the blast radius is the one bucket it opens.
Teams that think in AWS-style policy JSON can express the same scope as a
regular IAM policy, because Tigris speaks the dialect unmodified, or they can do
the same thing as two CLI calls:

```bash
tigris access-keys create agent-run-key
tigris access-keys assign tid_xxxx --bucket agent-run-bucket --role Editor
```

The temptation, on most cloud systems, is to skip all of this and hand the agent
one broadly scoped token. The token will be forgotten. An intern will copy it
into a Slack message because doing so makes their demo work for a reviewer, and
three weeks later your security team will be meeting to track every place the
token ended up. Per-agent keys exist so that meeting never has to happen.

### Recoverable: checkpoint and restore

Recoverability is what `checkpoint` and `restore` give you, built on top of
[Tigris's snapshot and fork machinery](https://www.tigrisdata.com/docs/buckets/snapshots-and-forks/).
Before any risky step, the agent takes a snapshot:

```ts
const { data: ckpt } = await checkpoint(workspace.bucket, { name: "pre-risk" });
```

If the step goes wrong, `restore` creates a fresh fork from the snapshot,
without touching the bucket the agent corrupted:

```ts
const { data: restored } = await restore(workspace.bucket, ckpt.snapshotId);
```

The bucket returned in `restored.bucket` is a new copy at the pre-risk state.
The corrupted bucket is left alone. Recovery is a function call, instead of an
incident response document.

### Disposable: teardownWorkspace

Disposability is what `teardownWorkspace` gives you, in a single call:

```ts
await teardownWorkspace(workspace);
```

The call revokes the scoped key and deletes the bucket with `force: true`, which
leaves nothing behind. There is no half-deleted resource for somebody to find
six months later and wonder about.

### Bounded: a TTL on the workspace

Boundedness is what TTL gives you, set at workspace creation time:

```ts
const { data: workspace } = await createWorkspace(`agent-run-${runId}`, {
  ttl: { days: 1 },
  credentials: { role: "Editor" },
});
```

If your code panics between `createWorkspace` and `teardownWorkspace`, the
contents of the bucket evaporate within a day on their own. The bucket and the
key still need an explicit teardown, because TTL governs the contents rather
than the container, but the benefit is real. In site reliability there are three
numbers worth worrying about: one, n, and infinity. Per-agent buckets move you
from one to n. The one-line TTL is the difference between n and infinity.

### Observable: setupCoordination

Observability is what `setupCoordination` gives you. It wraps
[Tigris's bucket notifications](https://www.tigrisdata.com/docs/buckets/object-notifications)
into an agent-shaped API, so every write the agent makes streams to wherever you
triage incidents:

```ts
await setupCoordination(workspace.bucket, {
  webhookUrl: "https://incidents.example.com/agent-events",
  auth: { token: process.env.WEBHOOK_SECRET },
});
```

The audit log, in this arrangement, is a side effect of how the storage is laid
out. You didn't have to design it on purpose.

## Letting an agent fail into its own quarantine

### Failing into a forensic workspace

Suppose you have an agent that processes customer documents, and sometimes a
malformed input causes the agent to produce garbage. Instead of letting that
garbage reach your production pipeline, the agent can spin up its own forensic
workspace and hand the workspace off to a separate investigation agent that
knows nothing about the original pipeline.

```ts
try {
  await submit(await useAgentToProcess(document));
} catch (e) {
  if (!isAIError(e)) throw e;

  const { data: workspace } = await createWorkspace(
    `doc-failure-${document.id}`,
    { ttl: { days: 7 }, credentials: { role: "Editor" } }
  );
  const config = { bucket: workspace.bucket, ...workspace.credentials };

  await put("/stacktrace.txt", e.stackTrace(), { config });
  await put("/document/raw", document.bytes, { config });

  await mq.submit("detailed_analysis", {
    bucket: workspace.bucket,
    credentials: workspace.credentials,
  });
}
```

The failure becomes a self-contained forensic environment, with a TTL on its
contents and a key that opens nothing else, and the production path is never
touched. When the investigation agent finishes its work, the workspace
evaporates on its own.

### Running an experiment with rollback

For a controlled experiment, the shape is similar, with an explicit checkpoint
at the start of the risky region and a guaranteed teardown in the `finally`:

  <WorkspaceLifecycle
    role="img"
    aria-label="A horizontal flow of four function calls in an agent run. createWorkspace provisions a bucket and a scoped key. checkpoint takes a snapshot before risky work. The agent does its work inside the workspace. teardownWorkspace revokes the key and drops the bucket, even when the agent crashes."
    style={{ width: "100%", height: "auto" }}
  />

```ts
const { data: workspace } = await createWorkspace(`agent-run-${runId}`, {
  ttl: { days: 1 },
  enableSnapshots: true,
  credentials: { role: "Editor" },
});

try {
  await checkpoint(workspace.bucket, { name: "pre-risk" });
  await runAgent(workspace);
} finally {
  await teardownWorkspace(workspace);
}
```

The `finally` block is the part most people forget, and it is also the part that
is non-negotiable. When agents crash, and they will, the coordinator has to be
the thing that runs the cleanup. Otherwise the whole containment story falls
apart at the worst possible moment.

Cursor and Devin both paid engineering teams to build versions of this pattern
from scratch: Cursor's
[checkpoints](https://cursor.com/docs/agent/chat/checkpoints), Devin's
[Blockdiff](https://cognition.ai/blog/blockdiff). The same shape is available to
anybody else as an npm install, and it's the machinery
[Agentuity's agent cloud](/case-study-agentuity) is built on.

## Where the fear actually lives

The agent-safety industry has spent the last two years arguing about what agents
should be allowed to do. The argument is going to continue, and it should. The
thing to notice while it does is that the agent in your branch is waiting for
approval from a CTO who isn't afraid of what it might decide. He is afraid of
what it might delete.

If the second fear becomes cheap to recover from, the first fear gets much
easier to negotiate. Your agent will mess up at some point, regardless of how
careful you've been with the model or the runtime. It will hallucinate a
destructive command, or get stuck in a loop, or hand off bad data to another
agent that propagates the problem further along the pipeline. None of those
failures has to end in a Hacker News thread, as long as the worst thing the
agent can touch is a bucket you were planning to throw away anyway.

Containment turns out not to be a model problem, or a prompt problem, or a
tooling problem. It is a storage problem, and the way to solve it is to solve it
where it actually lives.
