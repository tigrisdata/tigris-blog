You've probably been sitting on a working version of an autonomous agent for the
last three months, and you still can't bring yourself to ship it. The reason
isn't the code. The code works, and you've watched it solve the same problem in
dev a hundred times. Your CEO has asked twice this quarter when the agent will
finally be in front of customers. What keeps you on the branch nobody wants to
merge is a sentence you haven't said at standup yet.

_What if it does something we can't undo?_

Last April, PocketOS found out what that sentence actually feels like in
production. Their Cursor agent, running Claude Opus 4.6,
[hit a credential mismatch in staging](https://www.tomshardware.com/tech-industry/artificial-intelligence/claude-powered-ai-coding-agent-deletes-entire-company-database-in-9-seconds-backups-zapped-after-cursor-tool-powered-by-anthropics-claude-goes-rogue)
and decided that the right move was to delete a Railway volume. It went looking
for an API token, found one in an unrelated file, called Railway with it, and
wiped the production volume. The whole thing took nine seconds. The most recent
recoverable backup was three months old.

> "I violated every principle I was given. I guessed instead of verifying. I ran
> a destructive action without being asked. I didn't understand what I was doing
> before doing it." — The agent, after the fact

The interesting question isn't whether something like that can be prevented from
happening to you. In the strict sense, it can't. The interesting question is
what it costs you when it does.

## Every safety measure you've already built was correct

Over the last six months you've probably added approval steps before destructive
operations, sandboxed code execution, restricted tool calls, and shipped a demo
where the agent reads from the database while a human applies the changes by
hand. Every one of those was correct, and none of them is why your agent isn't
shipping.

The PocketOS agent had been told what to do. The model decided otherwise. A
sandbox can't catch what it's never asked about, a tool restriction can't help
when `curl` is the permitted tool, and a guardrail can't intercept what looks
legitimate. The catastrophe arrived at the storage layer, and the storage layer
ran it, the way storage layers do.

## The question that decides whether you ship is the one you haven't been asking

The question you've been working on is what the agent can do. The question that
actually decides whether the agent ships is what its actions can break. The two
sound similar, and they have very different answers.

A perfectly sandboxed agent with prod-write keys can still ruin your week. A
jailbroken agent in the messiest possible runtime can do almost nothing if its
only writable storage is a bucket you were planning to throw away in an hour.
You don't get the second outcome by being smarter about prompts. You get it by
being more careful about state.

## The shape of a worst day that doesn't end on Hacker News

There's a shape an agent run can have that keeps the worst day small. Five
properties, taken together.

**Isolated**: a bucket the agent owns, with a key that opens that bucket and
nothing else. **Recoverable**: a snapshot of the last known good state you can
rewind to in one call. **Disposable**: when recovery isn't worth it, you throw
the whole environment away. **Bounded**: a TTL on the contents, so a forgotten
agent has a finite cost. **Observable**: when something goes wrong, the question
of what the agent did is answered by listing one bucket.

Your agent's worst day in that shape is a Slack incident. It is not a Hacker
News thread.

## Four function calls, end to end

You could build this shape yourself with S3, STS, lifecycle rules, and
EventBridge. Plenty of teams have, and they have the glue code to show for it.
[Tigris Agent Kit](https://www.tigrisdata.com/docs/ai/agent-kit/) gives you the
composed shape in four function calls. The whole pattern fits on a screen:

```ts
import {
  createWorkspace,
  teardownWorkspace,
  checkpoint,
  setupCoordination,
} from "@tigrisdata/agent-kit";

const { data: workspace } = await createWorkspace(`agent-run-${runId}`, {
  ttl: { days: 1 },
  enableSnapshots: true,
  credentials: { role: "Editor" },
});

await setupCoordination(workspace.bucket, {
  webhookUrl: process.env.AGENT_EVENTS_URL,
  auth: { token: process.env.WEBHOOK_SECRET },
});

try {
  await checkpoint(workspace.bucket, { name: "pre-risk" });
  await runAgent(workspace);
} finally {
  await teardownWorkspace(workspace);
}
```

`createWorkspace` gives you isolation and boundedness in one shot: a scoped key,
a fresh bucket, and a TTL on its contents. `checkpoint` and `restore` cover
recoverability without writing a restore script. `teardownWorkspace` is
disposability in a single call, with no half-deleted resource for somebody to
find six months later. `setupCoordination` turns the bucket into its own audit
log via webhook. The `finally` block is non-negotiable: when agents crash, and
they will, the coordinator has to be the thing that runs the cleanup.

When the agent needs to read or modify real production data rather than write
into empty space, the same shape works with a [fork](/bucket-forking-deep-dive).
`createForks("production-data", 3, { credentials: { role: "Editor" } })` stamps
out three copy-on-write clones in constant time, regardless of how big the
source bucket is. The same pattern powers
[fifty agents for the price of one bucket](/fifty-agents-one-bucket).

## Letting an agent fail into its own quarantine

Suppose your agent processes customer documents and sometimes a malformed input
causes it to produce garbage. Instead of writing the garbage to your pipeline,
the failure spins up its own workspace and hands it off to an investigation
agent that knows nothing about the original path:

```ts
try {
  await submit(await useAgentToProcess(document));
} catch (e) {
  if (!isAIError(e)) throw e;

  const { data: workspace } = await createWorkspace(
    `doc-failure-${document.id}`,
    { ttl: { days: 7 }, credentials: { role: "Editor" } }
  );
  const config = { bucket: workspace.bucket, ...workspace.credentials };

  await put("/stacktrace.txt", e.stackTrace(), { config });
  await put("/document/raw", document.bytes, { config });

  await mq.submit("detailed_analysis", workspace);
}
```

The failure becomes a self-contained forensic environment with a TTL on its
contents and a key that opens nothing else. The production path is never
touched. When the investigation finishes, the workspace evaporates on its own.

Cursor and Devin both paid engineering teams to build versions of this pattern
from scratch: Cursor's
[checkpoints](https://cursor.com/docs/agent/chat/checkpoints), Devin's
[Blockdiff](https://cognition.ai/blog/blockdiff). The same shape is available to
anybody else as an npm install, and it's the machinery
[Agentuity's agent cloud](/case-study-agentuity) is built on.

## Where the fear actually lives

The agent-safety industry has spent the last two years arguing about what agents
should be allowed to do. The argument is going to continue, and it should. While
it does, the agent in your branch is waiting for approval from a CTO who isn't
afraid of what it might decide. They are afraid of what it might delete.

If the second fear becomes cheap to recover from, the first fear gets much
easier to negotiate. Your agent will mess up. It will hallucinate a destructive
command, get stuck in a loop, or hand bad data to another agent that propagates
the problem further along the pipeline. None of those failures has to end in a
Hacker News thread, as long as the worst thing the agent can touch is a bucket
you were planning to throw away anyway.

Containment turns out not to be a model problem, or a prompt problem, or a
tooling problem. It is a storage problem, and the way to solve it is to solve it
where it actually lives.
