## A contradiction, by design

Tigris is an S3-compatible object storage service with strong consistency within
a region and eventual consistency across regions, with sub-second replication
lag. A PUT returns; the next GET from the same region returns the object. A
DELETE returns; the next GET returns 404. Conditional operations evaluate
against the latest state. Those are the guarantees we sell, and they have to
hold under fault.

To hit the latency budget, every read goes through a Redis-backed edge cache in
front of FoundationDB metadata and S3-backed block storage. A cache hit returns
without ever touching FDB. Speed is what makes reads dangerous. Any
architecture that places a cache in front of a strongly consistent store is a
contradiction: the store guarantees that every read after a write reflects the
latest state, while the cache guarantees that most reads never touch the store.
Under clean operation, the two stay in sync because invalidations succeed and
events arrive in order. Under failure, they diverge.

This is the cache coherence problem, first studied in computer architecture[^mesi] and
inherited by every system that fronts a consistent store with a cache. Facebook
documented their version in their Memcache paper[^memcache]. Jepsen classifies
the user-visible symptom — a read returning a value that doesn't reflect a prior
completed write — as a stale read, a violation of strong
serializability[^jepsen-stale].

Antithesis[^antithesis] found four manifestations of this divergence in

angle on the same seam.

## What we were testing before, and what we weren't

Tigris runs a full three-region cluster in CI — not a single-cluster mock. Every
PR exercises integration tests, S3 compatibility tests, and Go's race detector,
against real regional gateways, servers, workers, FDB instances, caches, and
block storage. Tests write in one region and read from another, verifying
replication and read-after-write consistency end-to-end.

That caught a lot, but not these.

Two gaps: no fault injection, and scenarios authored by engineers. The first gap
meant CI only exercised clean paths — stable network, healthy regions,
responsive databases. The second meant coverage was bounded by what we thought
to write. The failure combinations that produce real consistency bugs — a region
slowing mid-replication, a retry arriving out of order, a Redis blip during an
async invalidation — weren't in the suite because they're combinatorial and hard
to articulate as tests.

We started with Antithesis in July 2025 specifically to close those gaps.
Antithesis runs the full system in a deterministic simulator under aggressive
fault injection, explores the state space autonomously, and lets you replay any
violation it finds. It's full-system property-based testing with faults.

Porting CI tests over took work. Integration tests are binary — operations pass
or fail. Under fault there's a third state: ambiguous. A write returns a
timeout; did it commit or not? A script that reads the timeout as "failure"
produces false negatives when the object is actually there. So the first job was
refactoring integration tests into fault-tolerant scripts (read, write,
read-then-write, delete, rename) that reason about all three outcomes. The
scripts compose: Antithesis orchestrates random sequences across hundreds of
thousands of scenarios, each under a different pattern of injected faults.

On top of the standard Antithesis fault catalog we added regional faults —
entire regions slowing or going offline, not just individual processes. That's
where the hard bugs surface, because losing a region means the
replication pipeline, cache layer, and regional servers all go down together,
and the system has to handle the combination.

Between July 2025 and March 2026 the setup ran 591 workloads, 73,178 virtual
hours, and explored roughly 20.3 million unique system states across 211,006
rollouts. Results land in Slack; triage converts into Linear. We review the
Antithesis runs daily. Antithesis's companion post covers the testing
methodology and the bug-finding journey in more detail.

## Four bugs, one seam

All four bugs Antithesis found share a single shape: a window between a metadata
operation succeeding in FDB and the edge cache reflecting that change. Under
normal conditions the window is closed by an invalidation or a deferred cleanup.
Under fault the window widens, and the cache and the metadata store go out of
sync.

### Bug 1: the delete-then-read race

The original delete path went through `deleteWithLock()`. The function acquired
a per-key cache lock to serialize cache population against concurrent reads,
called `deleteObjects()` which deleted the object from FDB, and relied on a
deferred cleanup step to evict the cache entry and write a tombstone. The
sequence was:

```
delete(key):
  1. acquire key lock
  2. delete from FDB            ← object is now gone
  3. (deferred) evict cache     ← runs after deleteObjects returns
  4. (deferred) write tombstone
  5. release key lock
```

Between step 2 completing and step 3 executing, a concurrent GET from another
request could hit the edge cache and return the now-deleted object with a 200
status. FDB said the object was gone. The cache still had it. From the user's
perspective: the DELETE succeeded, and the next GET returned the object they
just deleted.

Worse, if the gateway process crashed between the FDB delete and the deferred
eviction, the cache entry survived until its TTL expired — five minutes for
local objects, longer for objects cached from remote regions. During that window
every read returned a deleted object.

This is a linearizability violation[^linearizability]. Linearizability requires
that every operation appear to take effect atomically at a single point in real
time between its invocation and its response. A DELETE that returns and is
immediately followed by a GET that sees the deleted object cannot be placed on
any consistent real-time timeline. Jepsen classifies the observed symptom as a
stale read and treats it as a safety violation of strong
serializability[^strong-ser], not a performance regression. Facebook's Memcache
paper describes a structurally identical problem under the name "delete
consistency"[^memcache-34] and proposes lease tokens as the remedy — a
conceptually parallel mechanism to the tombstone barrier we describe below.

The Antithesis delete-then-get workload caught it on the first run. It issued a
DELETE followed by an immediate GET on the same key from regional endpoints; the
GET arrived in the few-millisecond window between step 2 and step 3 and returned
the deleted object. No engineer would have written that test, because no
engineer would have thought to test a millisecond-wide interleaving between two
steps of the same operation.

### Bug 2: rename treating cache invalidation as a hard dependency

A rename in Tigris is an atomic copy-then-delete at the metadata layer. FDB
commits the transaction; the system then distributes the change asynchronously —
replicate metadata to all regions, invalidate caches for the old key in every
region.

The old code treated the cache invalidation for the old key as a hard
requirement for the rename to succeed. A transient Redis failure caused the
invalidation to fail, which caused the rename to be reported as failed, which
triggered a rollback path on an operation that had already committed in FDB. A
best-effort downstream step was taking down a durable upstream operation.

The fix is structural: decouple the two. Rename succeeds when the FDB
transaction commits. Cache invalidation follows asynchronously. A transient
Redis failure can't block a durable metadata operation; the tombstone barrier on
the read path (below) handles the possibility of a stale cache hit in the
interim.

### Bug 3: replication events without timestamp comparison

Block replication events carry a `LastModified` timestamp. Under normal
conditions events arrive in order and the replication logic applies them in
sequence. Under Antithesis fault injection, events are delayed, retried, and
reordered in ways production rarely produces, and the replication code was not
comparing timestamps. An older version of an object could overwrite a newer one.

This is a last-writer-wins problem[^crdts], and the bug was that we never
enforced last-writer-wins. Ordering across components that don't share a clock
is the problem Lamport framed in 1978[^lamport]; in practice, any replication
layer that accepts events out of order needs an explicit timestamp comparison to
decide which wins. The fix was that: reject an incoming event whose
`LastModified` is older than the object already present. Newer data always wins,
regardless of event arrival order.

### Bug 4: strong consistency violation under regional failure

Under specific failure conditions involving partial regional faults, the read
path could observe a state that did not reflect a previously acknowledged write.
The fix tightened the read-path guarantees so the system validates state before
serving, even under degraded regional conditions. It's the least-bounded of the
four and a current focus; a fuller write-up lands with the
next round of component-level tests.

## The fix: reorder, then barrier

### Reorder

The first change is to the delete path itself. Instead of deleting from FDB
first and evicting the cache in a deferred cleanup, the code now evicts the
cache eagerly while holding the per-key lock, then deletes from FDB:

```
delete(key):
  1. acquire key lock
  2. evict cache entry          ← moved before FDB delete
  3. delete from FDB
  4. (deferred) evict again
     + write tombstone          ← defense-in-depth
  5. release key lock
```

The lock prevents a concurrent read from repopulating the cache during the
window. We swallow the eviction error on purpose: if Redis is temporarily
unreachable, the FDB delete still proceeds. The deferred eviction and tombstone
still run afterward as defense in depth. The alternative — fail the delete if
the cache eviction fails — would mean a Redis blip blocks every delete, which is
strictly worse than a brief window of stale cache bounded by TTL. FDB is the
authoritative operation; the cache is a performance optimization.

### Barrier

Reordering closes the race within a single operation. It does not handle the
case where a stale version arrives later through async replication and tries to
repopulate the cache. That's what the tombstone barrier is for.

A tombstone is a marker. Instead of removing a deleted key from the
cache, we replace it with an entry that says "this key was deleted at this
timestamp." On the read path, a tombstone tells the gateway to return 404
immediately without going to FDB. On the write path, a tombstone is a gate:

```
on cache_write(key, object):
  tombstone_ts = get_tombstone_timestamp(key)
  if tombstone_ts exists AND object.last_modified <= tombstone_ts:
    reject                       // stale data, do not cache
  else:
    write to cache
```

Every write to the cache — direct read-through, async replication from a
follower region, retry of a delayed event — passes through this check. If the
object being written is older than the tombstone, the write is rejected. That is
what makes the barrier work regardless of where the stale data came from.

The tombstone itself has a TTL. Without one, the cache fills with markers for
keys that will never be queried again. The TTL has to outlast any plausible
replication delay, retry window, or async event that might deliver a stale
version of the deleted object. Too short and a delayed event clears the barrier
and re-caches a stale copy; the loop restarts. Getting this right depends on
worst-case replication lag under failure conditions, which is hard to reason
about without testing under faults. We run with five minutes for local objects
and longer for remote.

Three alternatives we considered and rejected:

- **ETag check on every cache hit.** Requires an FDB round-trip per read, which
  defeats the cache.
- **Shorter TTLs.** Shrinks the window without closing it, and
  costs hit rate across every object to solve a delete-specific problem.
- **Move the deferred eviction earlier, but still after the FDB delete.**
  Doesn't close the race. A concurrent GET can still win between FDB returning
  and the eviction executing.

The barrier is source-agnostic, failure-tolerant, and applies to every write
path — not just deletes. Renames, overwrites, any mutation that writes a
tombstone gets the same protection.

## Three layers, one property

The general pattern that came out of the audit is a three-layer defense. Each
layer catches a class of failure the one above cannot:

- **Write path: eager invalidation.** Every mutation invalidates the cache entry
  before the mutation commits to the primary store, not after. Any gap between
  "the source of truth changed" and "the cache knows about it" is a window for a
  concurrent read to serve stale data. Eager eviction under the key lock closes
  it.
- **Read path: tombstone barriers.** Eager invalidation handles the case where
  the cache is reachable and the eviction succeeds. It doesn't handle the case
  where a stale version shows up later through async replication and tries to
  repopulate the cache. The tombstone is a read-time check, not a write-time
  guarantee. Even if the eviction fails, even if Redis is unreachable, even if a
  follower delivers a stale copy, the tombstone blocks the repopulation.
- **Replication boundary: timestamp ordering.** The first two layers handle
  divergence within a region. Across regions, replication events arrive
  asynchronously and can be delivered out of order or retried. Without an
  explicit timestamp comparison on incoming events, an older version can
  overwrite a newer one. The effect on cache coherence is the same: the data
  behind the cache goes backward, and the cache serves it. Spanner handles the
  analogous problem at the database layer with TrueTime and external
  consistency[^spanner]; the mechanism is different but the requirement is the
  same.

Each layer individually was already in the system. Each layer individually
passed its tests. What we audited, and what the fixes apply to, is every code
path where the three layers interact — the places where a failure in one layer
has to be caught by the next. That is the class of coverage CI wasn't giving us,
and the class of coverage Antithesis is built for.

The edge cache can no longer serve a stale entry after a successful mutation,
regardless of whether the eager eviction succeeded, whether Redis was reachable
at delete time, or whether replication events arrive out of order. The cache is
still a performance optimization. It just can't violate consistency.

We haven't encountered a cache coherence bug since the reorder and barrier
landed.

## What's next

Our Antithesis setup exercises the system end-to-end through the S3 API, which
is where all four of these bugs surfaced. The next step is component-level
testing — putting individual subsystems under Antithesis in isolation, where the
fault surface is narrower and the invariants are more specific.

Three targets:

- **Replication pipeline.** Metadata distribution, block distribution, and cache
  invalidation run independently. Under faults, the gaps between them widen, and
  the ordering and delivery assumptions between the paths are where the next
  class of bugs is likely to live.
- **Async processing layer.** Background processes handle replication delivery,
  cache invalidation, tombstone cleanup, and lifecycle management. When one of
  them fails mid-operation, recovery has to avoid leaving partially-completed
  work and has to avoid duplicating side effects. Those are hard properties to
  verify with scripted tests because the failure timing matters as much as the
  failure itself.
- **Regional failover.** We support single-region, dual-region, multi-region,
  and global buckets, each with different consistency and availability
  properties. With regional faults added to the Antithesis setup, the next
  question is whether service continues from remaining regions, whether
  replication catches up cleanly on recovery, and whether all regions converge
  to the same state afterward.

[^mesi]:
    Hennessy, J. L., & Patterson, D. A., _Computer Architecture: A Quantitative
    Approach_, Chapter 5 on cache coherence protocols. The MESI protocol is the
    canonical hardware-level treatment. The distributed systems problem is the
    same invalidation-and-coherence problem without the luxury of a shared bus.

[^memcache]:
    Nishtala, R., et al., "Scaling Memcache at Facebook," NSDI 2013. See
    especially Section 3.3 on delete consistency.

[^jepsen-stale]:
    Kingsbury, K., "Consistency Models: Stale Read," Jepsen.
    [https://jepsen.io/consistency/phenomena/stale-read](https://jepsen.io/consistency/phenomena/stale-read).
    A stale read is a safety violation of strong serializability, not a
    performance issue.

[^antithesis]:
    Antithesis is a deterministic simulation testing platform. The system under
    test runs in a hypervisor-level simulator that controls scheduling, network,
    disk, and time; faults are injected deterministically, the state space is
    explored autonomously, and any violation found is exactly replayable.

[^linearizability]:
    Herlihy, M. P., & Wing, J. M., "Linearizability: A Correctness Condition for
    Concurrent Objects," ACM TOPLAS 1990. The formal definition of
    linearizability and its real-time ordering constraint.

[^strong-ser]:
    "Consistency Models: Strong Serializable," Jepsen.
    [https://jepsen.io/consistency/models/strong-serializable](https://jepsen.io/consistency/models/strong-serializable).
    Strong serializability combines serializability with linearizability:
    transactions appear to execute in some serial order, and that order respects
    real time.

[^memcache-34]:
    Nishtala et al. 2013, Section 3.3. Facebook's lease-token mechanism gives
    the cache a way to reject a stale write that arrives after a delete —
    structurally parallel to the tombstone barrier.

[^crdts]:
    Shapiro, M., Preguiça, N., Baquero, C., & Zawirski, M., "Conflict-free
    Replicated Data Types," INRIA Research Report 7687, 2011. The LWW-register
    is the canonical construction; it requires a total order on timestamps to
    resolve concurrent writes deterministically.

[^lamport]:
    Lamport, L., "Time, Clocks, and the Ordering of Events in a Distributed
    System," Communications of the ACM, 1978.

[^spanner]:
    Corbett, J. C., et al., "Spanner: Google's Globally-Distributed Database,"
    OSDI 2012. Section 4 introduces TrueTime and external consistency. The
    mechanism is a bounded-uncertainty clock; the requirement it solves for —
    reads reflecting the latest committed state when the read path bypasses the
    serialization point — is structurally the same as ours at the cache layer.
